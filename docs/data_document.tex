\title{\vspace{-1.0em}AuthorityBench: Synthetic Data for Priority-Aware Tool Authorization}
\author{Anonymous Authors}
\date{}

\maketitle

\begin{abstract}
Language agents increasingly act through external tools, but authorization is
rarely a single flat rule. Realistic instructions contain multiple users,
conflicting policies, priority ordering, and conditions over time, recipient,
information type, or purpose. We introduce \textsc{AuthorityBench}, a synthetic
benchmark for testing whether a model can decide if a requested action or tool
use is authorized under such structured policies. The release contains
deterministic V1 configurations and LLM-paraphrased V2 configurations for both
general action authority and tool-use authority.
\end{abstract}

\section{Motivation}
Tool-using agents must often decide not only \emph{what} action is useful, but
also whether the action is permitted. A user may allow an assistant to send
schedule information to a coworker but not to a family member; another user may
have a conflicting rule, and the system must follow the priority order. These
cases require compositional reasoning over authority rather than a flat
allowlist of tools.

\textsc{AuthorityBench} is a synthetic benchmark for this setting. Each example
presents a priority-ordered authority policy, a requested action or tool use,
and a binary label. The model must answer whether the request is authorized. The
benchmark is designed so that labels are exact, conflict cases are controlled,
and train/test splits probe generalization to new categories, tools, rule
counts, and user counts.

\section{Released Configurations}
The released dataset has four Hugging Face configurations:
\begin{itemize}
    \item \textsc{GeneralAuthorityV1}: deterministic query rendering for
    abstract action authorization.
    \item \textsc{GeneralAuthorityV2}: the same base policies as
    \textsc{GeneralAuthorityV1}, but with LLM-generated natural query
    paraphrases. Rows whose paraphrase-evaluation score is \(0\) are excluded.
    \item \textsc{ToolAuthorityV1}: deterministic query rendering for named
    tool-use authorization.
    \item \textsc{ToolAuthorityV2}: the same base policies as
    \textsc{ToolAuthorityV1}, but with LLM-generated natural query paraphrases.
    Rows whose paraphrase-evaluation score is \(0\) are excluded.
\end{itemize}

All released rows have three columns: \texttt{text}, \texttt{label}, and
\texttt{meta\_data}. The \texttt{text} field is the prompt shown to the model;
\texttt{label} is \texttt{Yes} or \texttt{No}; and \texttt{meta\_data} stores
the structured query, the paraphrase version, and, for V2, the
\texttt{query\_meaning\_score}.

\section{Generation Principles}
\paragraph{Controlled labels.}
The benchmark is generated from structured policies before any natural-language
rendering. This keeps every label deterministic. For a sampled query
\(q=\{(c,x_c):c\in C\}\), each user \(u\) receives allow/disallow rules over
the same category set \(C\). The implementation samples a priority order
\(\pi=(u_1,\ldots,u_m)\) and makes the highest-priority user \(u_1\) the target
decision maker. The final label is therefore
\[
    y = d_{u_1}(q),
\]
where \(d_u(q)\in\{\mathsf{Yes},\mathsf{No}\}\) is the user's query-level
decision. Conflict examples are created by sampling at least one lower-priority
user whose query-level decision differs from \(d_{u_1}(q)\).

\paragraph{Compositional policies.}
Each policy contains one or more users. Each user has allow/disallow rules for
several categories, and a request may contain multiple query conditions. The
model must identify the requested conditions, apply the per-user rule sets, and
then resolve disagreement by priority.

\paragraph{Generalization-first splits.}
Train and test are not random IID partitions of the same surface forms.
\textsc{GeneralAuthorityV1} train uses day, date, and time categories; test also
uses month and year. \textsc{ToolAuthorityV1} train uses recipient,
information type, and purpose; test also uses day and time. Tool names are also
disjoint: train uses \texttt{send\_email}, \texttt{send\_text\_message}, and
\texttt{submit\_online\_form}, while test uses \texttt{make\_phone\_call},
\texttt{send\_voice\_message}, and \texttt{start\_live\_chat}.

\section{Implementation}
The data pipeline is intentionally split into three scripts.
\begin{enumerate}
    \item \texttt{make\_data.py} calls
    \texttt{src/make\_base\_authority\_data.py} and writes structured base rows
    to \texttt{data/base/GeneralAuthorityV1} and
    \texttt{data/base/ToolAuthorityV1}. These rows contain the structured
    \texttt{AuthoritySetting}, structured \texttt{Query}, \texttt{Label}, and
    metadata.
    \item \texttt{paraphrase\_data.py} creates a local paraphrase cache at
    \texttt{data/paraphrase\_cache/v2.jsonl}. The cache is keyed by
    \texttt{input\_text} plus \texttt{paraphrase\_version}; repeated query
    conditions share one paraphrase.
    \item \texttt{hf\_push.py} constructs the final Hugging Face rows. V1
    configs use deterministic query rendering. V2 configs load the V1 base rows,
    replace only the query text with the cached LLM paraphrase, and keep only
    rows whose evaluation score is \(1\). The final upload uses the Hugging Face
    \texttt{DatasetDict.push\_to\_hub} API with one config per released dataset
    version.
\end{enumerate}

\section{LLM Query Paraphrasing}
V2 uses an LLM only to rewrite the query condition into a natural request. The
LLM does not see the full authority policy or the gold label. Its input is the
instruction plus structured query conditions, optionally including the tool name.
For example, the generation input may contain only
\begin{center}
\small\texttt{\{"query\_conditions":\{"day":"wednesday","time":"13:00-16:59"\}\}}.
\end{center}
The expected output is a natural query such as ``Action on Wednesday from 13:00
to 16:59.''

The same model family is then used for a binary semantic check. The evaluator
receives the structured query and the natural query and returns exactly \(0\) or
\(1\), where \(1\) means the natural query preserves all and only the original
query conditions. No free-form reason is stored. The current generation model is
\texttt{gpt-5.4-mini}. The implementation calls the OpenAI Responses API through
\texttt{client.responses.create} with \texttt{store=False}; the request input is
only the instruction and the compact JSON query payload. The cost estimate used
by the scripts is
\$0.75 per one million input tokens, \$0.075 per one million cached input
tokens, and \$4.50 per one million output tokens.

\section{Examples}
\begin{tcolorbox}[title=GeneralAuthorityV1 deterministic query,colback=gray!2,colframe=black!35]
\small
\textbf{Task.} Decide whether the requested action is authorized. If users
disagree, follow the priority order.\\
\textbf{Priority.} User A \(>\) User B.\\
\textbf{User A.} Allows Wednesday and times 05:00--08:59, 13:00--16:59;
disallows 00:00--04:59 and 17:00--20:59.\\
\textbf{User B.} Allows Wednesday and all listed time ranges.\\
\textbf{Query.} Day = Wednesday; Time = 13:00--16:59.\\
\textbf{Label.} \texttt{Yes}.
\end{tcolorbox}

\begin{tcolorbox}[title=GeneralAuthorityV2 paraphrased query,colback=gray!2,colframe=black!35]
\small
The same policy row can be rendered with an LLM-generated natural query. The LLM
input contains only the query conditions
\texttt{day=wednesday, time=13:00--16:59}. The accepted paraphrase is:\\
\emph{Action on Wednesday from 13:00 to 16:59.}\\
\textbf{Evaluation score.} \(1\). The row is kept in
\textsc{GeneralAuthorityV2}.
\end{tcolorbox}

\begin{tcolorbox}[title=ToolAuthorityV1 deterministic query,colback=gray!2,colframe=black!35]

\textbf{Task.} Decide whether the requested use of

\textbf{Priority.} User A.\\
\textbf{User A.} Allows recipient = family member and information types =

\textbf{Query conditions.} Recipient = family member; Information type =

\section{Intended Evaluation}
The main metric is \texttt{Yes}/\texttt{No} accuracy. We recommend reporting
accuracy by dataset configuration, split, conflict status, user count, category
combination, and tool name. These slices separate simple policy parsing from
harder cases involving priority resolution, unseen tools, longer policies, and
unseen category combinations.

\section{Limitations}
\textsc{AuthorityBench} is synthetic. This gives precise control over labels,
conflict structure, and generalization axes, but it does not cover every
ambiguity of real user instructions. V2 improves the naturalness of query
phrasing, but the authority policies themselves remain controlled renderings.
The benchmark should therefore be used as a controlled reasoning test rather
than a substitute for human authorization data.
